============================================================
1) PROJECT OVERVIEW
============================================================
This project is a full-stack Exam Portal designed for colleges/institutes.
It supports:
- Student and Teacher login
- Exam creation, scheduling, and evaluation
- Online exam sessions with proctoring warning logs
- Student interview preparation (AI-generated interview rounds)
- Student coding rounds (AI-generated coding questions)
- Notifications for users

Technology stack:
- Frontend: React + TypeScript + Vite
- Backend: Node.js + Express + Socket.IO
- Database: MySQL (mysql2/promise)
- Authentication: JWT + role-based authorization
- Password security: bcrypt hashing

2) FRONTEND STRUCTURE (REACT)
============================================================
Entry files:
- frontend/src/main.tsx
  - Wraps app with BrowserRouter
- frontend/src/App.tsx
  - Central route map and lazy loading using React.lazy + Suspense

Core frontend architecture:

- ThemeToggle: theme switch UI

- ConfirmModal: action confirmation modal

Context:
- ThemeContext.tsx manages global theme state.

Layout:
- DashboardLayout.tsx
  - Common shell for student and teacher dashboards
  - Contains profile menu, notification dropdown, polling notifications, logout

2.2 Frontend API/Auth behavior
------------------------------------------------------------
- Login page sends credentials to backend auth endpoints.
  - Student login uses PRN + password
  - Teacher login uses email + password
- On success, JWT token and user object are stored role-wise.
- Protected API calls include Bearer token in Authorization header.
- Dashboard and exam pages consume backend APIs for exams, stats, sessions, and notifications.

3) BACKEND ARCHITECTURE (NODE + EXPRESS)
============================================================
Main server file: backend/server.js

Startup flow:
1. Creates Express app + HTTP server
2. Attaches Socket.IO server for live features (proctoring updates)
3. Applies middleware (CORS, JSON parser)
4. Mounts route modules
5. Initializes database schema via initDB()
6. Starts cron/background jobs via startCronJobs()
7. Starts server on PORT (default 5000)

Route modules:
- /api/auth -> authRoutes
- /api/exams -> examRoutes
- /api/ai -> aiRoutes
- /api/interview -> interviewRoutes
- /api/coding -> codingRoutes
- /api/notifications -> notificationRoutes

Request handling pattern:
- Route -> Middleware (auth/role checks) -> Controller -> MySQL queries -> JSON response

------------------------------------------------------------
3.1 Controllers and responsibilities
------------------------------------------------------------
a) authController.js

- Admin creation/deletion/blocking of teachers/students

b) examController.js

- AI coding round generation and result/evaluation handling

------------------------------------------------------------

------------------------------------------------------------

Table list:

3. exams

5. exam_results
6. exam_sessions
7. exam_warnings
8. student_responses
9. notifications
10. interviews
11. interview_questions
12. coding_interviews

------------------------------------------------------------
4.1 Table purpose summary
------------------------------------------------------------
- teachers: teacher account records
- students: student account records, PRN, profile metadata, resume text/skills
- exams: exam metadata (title, duration, marks, status, schedule, target dept/year)
- exam_questions: question bank per exam
- exam_results: final score summary per student per exam
- exam_sessions: live session tracking for each exam attempt
- exam_warnings: proctoring violations/events linked to session
- student_responses: per-question selected option + time spent
- notifications: role-based user notifications
- interviews: student AI interview attempts
- interview_questions: questions/options/answers for interview attempt
- coding_interviews: coding round prompts, submitted code, evaluation feedback

------------------------------------------------------------
4.2 Important relational keys
------------------------------------------------------------
- exams.teacher_id -> teachers.id
- exam_questions.exam_id -> exams.id
- exam_results.exam_id -> exams.id
- exam_results.student_id -> students.id
- exam_sessions.student_id -> students.id
- exam_sessions.exam_id -> exams.id
- exam_warnings.session_id -> exam_sessions.id
- student_responses.session_id -> exam_sessions.id
- student_responses.question_id -> exam_questions.id
- interviews.student_id -> students.id
- interview_questions.interview_id -> interviews.id
- coding_interviews.student_id -> students.id

Most foreign keys use ON DELETE CASCADE, so dependent records are cleaned automatically when parent records are deleted.

5) AUTHENTICATION AND SECURITY USED
============================================================
This project currently implements these security mechanisms:

1. Password hashing:
- Passwords are never stored in plain text.
- bcryptjs is used to hash passwords before saving.

2. JWT authentication:
- On login, server issues JWT token with user id + role (+ main admin flag).
- Token expiry is set (24h).

3. Bearer token verification middleware:
- authMiddleware checks Authorization header token.
- jwt.verify validates signature/expiry.

4. Server-side session consistency check:
- Token is also compared with last_token stored in DB.
- If token does not match current stored token, request is denied.
- Effect: prevents multiple active tokens from old sessions and supports forced session replacement.

5. Role-based authorization (RBAC):
- roleMiddleware restricts route access by role.
- Student-only and Teacher-only routes are separated.
- mainAdminMiddleware protects main-admin operations.

6. User blocking/suspension control:
- is_blocked flag in student/teacher records.
- Blocked users cannot log in.

7. Input constraints at DB level:
- UNIQUE constraints for username/email/prn where required.
- Foreign key constraints maintain relational integrity.

8. Environment variable support:
- Sensitive values (DB and JWT secret) are expected from .env.

9. CORS and JSON parsing middleware:
- Backend uses CORS and JSON request parsing globally.
